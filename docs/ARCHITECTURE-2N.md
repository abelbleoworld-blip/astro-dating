# ARCHITECTURE-2N — манифест шкалы степеней двойки

**Статус:** принят 2026-04-17. Пилот: 1 неделя.
**Применимость:** astro-dating (первичный), Abel Ecosystem (через mesh-memory).
**Размер этого файла:** ~256 строк (2⁸ — модуль документации).

## 1. Принцип

Стабильная конфигурация системы имеет размер вида **2ⁿ × 2ⁿ** или **2ⁿ единиц**. Каждый уровень содержит 4 копии предыдущего — это **фрактальная рекурсия**. На каждом уровне работает одна и та же логика «4 групп по 4».

| Уровень | Размер | Что | Пример из проекта |
|---|---|---|---|
| 2² | 4 | базовый кластер | `src/` `data/` `results/` `docs/` |
| 2⁴ | 16 | модуль | 16 якорей сейчас (стабильно ✅) |
| 2⁶ | 64 | система | 64 якоря (цель), 64 строки в файле |
| 2⁸ | 256 | большой модуль | этот файл, Python-скрипт |
| 2¹⁰ | 1024 | корпус-статья | финальный peer-review paper |
| 2¹² | 4096 | книга | будущая монография |
| 2¹⁴ | 16384 | библиотека | весь Abel Ecosystem |
| 2¹⁶ | 65536 | свод | предел личной системы |

## 2. Почему это работает

Стабильность 64 в природе и культуре — не совпадение:
- ДНК: 4 базы × 3 позиции = 64 кодона (биохимия)
- И-Цзин: 2 (Инь/Ян) × 6 линий = 64 гексаграммы (китайская традиция)
- Шахматы: 8×8 = 64 клетки (геометрия игры)
- ASCII: 2⁸ = 256 (информатика)
- CPU слово: 64 бит (архитектура железа)
- TCP/IP пакет: 4 КБ = 4096 байт (сетевой стек)

Когнитивная подсказка: мозг легко удерживает 4 элемента (рабочая память) и иерархию из 6 уровней (Miller 7±2). Это даёт естественный потолок ~64 = 4⁶ = 2⁶.

Архитектурное: степени двойки оптимальны для деления, перемещения, кэширования, индексации. Любая ИТ-система внутри устроена так. Применяя к контенту/проектам — наследуем те же свойства.

## 3. Три свойства интегральности

**Самоподобие.** Открыл якорь S1 (64 строки) → видишь 4 секции по 16. Открыл проект (16 якорей) → видишь 4 группы по 4. Открыл библиотеку (16 проектов) → видишь 4 кластера по 4. Одна когнитивная нагрузка на каждом масштабе.

**Перевод вверх и вниз.** Любой элемент уровня N может быть «развернут» в 16 на уровне N-1, или «свёрнут» в 1/16 на уровне N+1. Якорь S1 (64 строки) → глава статьи (1/16 от 1024). Статья → глава книги (1/4 от 4096). Никакая работа не пропадает.

**Стабильные точки выхода.** Проект «готов» на каждом уровне 2ⁿ × 4: 16, 64, 256, 1024. Между этими точками — недосказанное состояние, нельзя останавливаться. На точках — можно публиковать, передавать, объявлять готовым.

## 4. Шаблон 64-строчного якоря

Образец: `docs/anchors/S1-iching-dna.md`. Принят 2026-04-17 как канон.

Структура: ровно 64 строки в файле, 4 секции по 16 строк. Заголовок секции работает как разделитель, пустые строки между секциями не используются.

| Секция | Строки | Содержание |
|---|---|---|
| 1. Тезис | 1-16 | заголовок ## + что и зачем + таблица + связи с другими якорями |
| 2. Метод | 17-32 | заголовок ## + 4 шага проверки + статистика + протокол A↔B |
| 3. Литература и ограничения | 33-48 | заголовок ## + 5 источников + 4 риска + 3 митигации |
| 4. TODO и cross-links | 49-64 | заголовок ## + код + документы + ↑↓↔ ссылки + дата ревью |

Каждая секция = 1 заголовок + 15 строк содержания. Каждая строка = одна мысль или один пункт списка. Никаких длинных параграфов — формат идёт от структуры, не от потока сознания.

Применить к **48 будущим якорям** до достижения цели 64. Существующие якоря в `results/*.md` — постепенно подровнять (не приоритет, делаем по мере касания).

## 7. Что НЕ делать (anti-patterns)

- ❌ Не оставлять проект на «промежуточном» числе (32 якоря, 100 строк, 7 модулей) — нестабильное состояние. Если уж оставляешь — оформляй как «частично закрытый модуль 16, плюс 16 черновиков».
- ❌ Не растягивать содержание ради числа (64 строки воды вместо 32 строк смысла). Лучше 32, чем 64 раздутых. Если контент не дотягивает — секция короче, добавляются placeholder-строки `- [ ] TODO: расширить пункт X`.
- ❌ Не сжимать содержание вырезая суть (вмещать 200 строк в 64 потерей нюансов). Если не лезет в 64 — это сигнал, что нужно разделить на 2 якоря по 64 каждый.
- ❌ Не применять принцип к областям где он не работает. Юридический документ требует своей структуры (статьи, пункты по нумерации закона). Поэтический текст — своей. Шкала 2ⁿ — для **систем**, не для всего.
- ❌ Не превращать в религию. Если **очень** нужно 17 якорей вместо 16 — сделай 17 и пометь «стремимся к 32». Принцип — гид, не тюрьма.

## 8. Триггеры пересмотра принципа

Если хотя бы одно из:
- На промежуточном числе (32 якоря) ощущение «правильное, не хочу растягивать» — пересмотр, может 2ⁿ не универсально.
- Шаблон 64-строки ощущается тесным/сухим систематически — попробовать 2 секции по 32 или 8 секций по 8.
- Есть конкретный проект (не astro-dating), где шкала очевидно ломается — фиксируем как ограничение применимости.
- Через 3 месяца применения метрика «продуктивность × удовольствие» ниже чем без принципа — отказ.

Триггер усиления:
- Если все 64 якоря собрались легко и красиво — следующая шкала 256 (большой проект).
- Если 8×8 mesh-распределение работает играючи — расширяем до 16×16=256 (16 машин, 16 ролей — это уже команда).
